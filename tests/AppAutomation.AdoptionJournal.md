# Журнал внедрения AppAutomation

## Область
- Репозиторий-потребитель: `SkillChat`
- Тестируемое приложение (AUT): `SkillChat.Client` (`Avalonia`, `net10.0`)
- Дата внедрения: `2026-03-24`
- Цель: интегрировать `AppAutomation`, добавить детерминированные смоук-тесты и зафиксировать шероховатости при реальном онбординге потребителя.

## Запись 1. Предварительное исследование
- Шаг:
  - Изучены `README.md`, `quickstart.md`, `adoption-checklist.md`, `project-topology.md`, `advanced-integration.md`.
- Что сработало:
  - Канонический потребительский сценарий становится понятным, если читать документацию в связке.
  - Разделение на `Authoring`, `Headless`, `FlaUI` и `TestHost` описано ясно.
- Шероховатости:
  - Рекомендации по версиям расходятся между официальными поверхностями репозитория:
    - `README` репозитория в разделе `Fast Path` всё ещё ссылается на `1.1.0`;
    - список релизов GitHub показывает последнюю версию `1.2.0` от `2026-03-18`;
    - `docs/appautomation/quickstart.md` в ветке `master` ссылается на `2.1.0`.
  - Потребителю приходится самому делать вывод, какая версия считается канонической, ещё до запуска первой команды.
- Предложения:
  - Оставить в репозитории один источник истины по версии и генерировать из него все примеры установки.
  - Добавить рядом с `Fast Path` короткую заметку вида: если документация и релизы расходятся, доверять `X`.
  - Публиковать компактную матрицу совместимости и версий прямо в `README`, а не только в более глубоких документах.

  - Добавить заметный раздел с предварительными требованиями, где явно указан нужный источник пакетов, если пакеты недоступны на `nuget.org`.

## Запись 6. Проверка `doctor` и сгенерированного каркаса
- Шаг:
  - Сгенерирована каноническая структура командой:
    - `dotnet new appauto-avalonia --name SkillChat --AppAutomationVersion 1.2.0 --TestTargetFramework net10.0 --FlaUiTargetFramework net10.0-windows7.0`
  - Выполнена команда:
    - `dotnet tool run appautomation doctor --repo-root .`
  - Сгенерированные проекты добавлены в `SkillChat.sln`.
  - Сгенерированные проекты собраны по отдельности.
- Что сработало:
  - Шаблон создал ожидаемую каноническую структуру.
  - `doctor` сразу завершился успешно с `0 error(s), 0 warning(s)`.
  - Существующего `SkillChat.Client/nuget.config` оказалось достаточно для `doctor`; дополнительный корневой `NuGet.Config` не понадобился.
  - Сгенерированные проекты успешно собираются после добавления в решение.
- Шероховатости:
  - Документация и чеклисты для потребителя создают сильное впечатление, что нужен `NuGet.Config` именно на уровне корня, тогда как фактическое поведение `doctor` мягче и полезнее, чем следует из формулировок.
  - Шаблон намеренно оставляет потребителя в состоянии с большим количеством заглушек, поэтому первая успешная сборка не означает "уже можно писать тесты"; эту границу стоило бы обозначить явнее.
- Предложения:
  - Уточнить в документации, что `doctor` принимает корректную конфигурацию NuGet как на уровне репозитория, так и во вложенных каталогах, и не требует файл строго в корне.
  - Добавить короткую заметку о том, что сгенерированная структура является лишь компилируемым каркасом, пока не заменены заглушки `TestHost` и селекторы AUT.

## Запись 7. Интеграция детерминированного бутстрапа
- Шаг:
  - Общий клиентский бутстрап вынесен из `Program`/`App`.
  - Добавлено разрешение пути к настройкам через `SKILLCHAT_SETTINGS_PATH`.
  - `SkillChat.AppAutomation.TestHost` настроен на:
    - реальные пути к решению и проекту;
    - изолированный временный `Settings.json`;
    - фабрику headless-окна через общий бутстрап.
  - Собраны `SkillChat.Client`, `SkillChat.AppAutomation.TestHost`, `SkillChat.UiTests.Headless`, `SkillChat.UiTests.FlaUI`.
- Что сработало:
  - AUT теперь можно инициализировать через одну общую точку бутстрапа вместо дублирования кода инициализации внутри тестов.
  - `TestHost` умеет запускать реальный `MainWindow` с тестовыми настройками.
  - Перенос определения IP и сведений о рантайм-среде из конструктора в `ConnectCommand` убирает лишний стартовый побочный эффект для смоук-тестов.
- Шероховатости:
  - Обычный `dotnet build` сначала падал, потому что `Avalonia.Designer.HostApp` из IDE держал блокировку на `SkillChat.Client\bin\Debug\net10.0`.
  - С точки зрения потребителя это означает, что путь десктопного запуска и сборки чувствителен к инструментам дизайн-тайма, которые держат открытым каталог вывода AUT.
  - В `AvaloniaDesktopLaunchHost` нет очевидного хука `AppAutomation`, который позволял бы перед запуском перенаправить результаты сборки во временный изолированный каталог.
- Предложения:
  - Задокументировать для потребителей Avalonia пограничный случай "закройте Avalonia designer / design host перед проверкой десктопного рантайма".
  - Рассмотреть возможность задавать пользовательские MSBuild-свойства или изолированные выходные каталоги в параметрах десктопного запуска, чтобы десктопная автоматизация была меньше связана со стандартным каталогом `bin` AUT.

## Запись 8. Написание headless-смоука на реальном экране Avalonia
- Шаг:
  - Добавлены минимальные якоря `AutomationProperties.AutomationId` для реального сценария `login/register`.
  - Текстовые ссылки, реагирующие только на нажатие, заменены на семантические `Button` при сохранении той же визуальной роли.
  - `HeadlessSessionHooks` реализован через:
    - `HeadlessUnitTestSession.StartNew(SkillChatAppLaunchHost.AvaloniaAppType)`
    - `HeadlessRuntime.SetSession(...)`
  - Написан детерминированный смоук-путь, который покрывает:
    - запуск на странице логина;
    - переход на страницу регистрации;
    - активацию кнопки отправки через переключение согласия;
    - заполнение полей регистрации;
    - возврат на страницу логина.
  - Выполнена команда:
    - `dotnet test --project tests\\SkillChat.UiTests.Headless\\SkillChat.UiTests.Headless.csproj -c Debug`
- Что сработало:
  - После регистрации глобальной headless-сессии в хуках `DesktopAppSession.Launch(...)` и `HeadlessControlResolver` заработали против реального `MainWindow`.
  - Каноническое разделение `Authoring -> Headless` хорошо ложится на смоук-сценарий, принадлежащий потребителю.
  - UI-автоматизация выявила реальный баг AUT, связанный с потоками, в `MainWindow` (`ScrollViewer.Offset` затрагивался вне UI-потока), то есть фреймворк полезен как ранний детектор проблем интеграции, а не только как исполнитель селекторов.
- Шероховатости:
  - Сгенерированный `HeadlessSessionHooks.cs` содержит только заглушку TODO. Потребителю всё ещё приходится самостоятельно находить точную обязательную пару API (`HeadlessUnitTestSession.StartNew` + `HeadlessRuntime.SetSession`) через эксперименты, чтение исходников или падение рантайма.
  - Первое сообщение об ошибке рантайма было полезным (`Headless session is not initialized...`), но оно появилось только после запуска теста. Это требование к настройке можно было бы задокументировать или зашить прямо в каркас.
  - `Button.Content` / `TextBlock.Text` в этом headless-сценарии Avalonia не использовались надёжно через `WaitUntilNameEquals` / `WaitUntilNameContains`, пока не были явно назначены значения `AutomationProperties.Name`.
  - Динамический текст метки (`RegisterErrorLabel`) не оказался стабильной поверхностью для проверки этого смоук-пути через automation-`Name`, поэтому потребителю пришлось сузить тест до переходов и изменений состояния вместо проверки встроенного сообщения об ошибке.
  - Сгенерированный раннер TUnit/Microsoft Testing Platform не принимает классический сценарий `dotnet test --filter ...`; он ожидает собственные механизмы фильтрации (`--treenode-filter` / `--filter-uid`). Это неожиданно при отладке, если документация показывает только общий `dotnet test`.
  - Во время написания тестов члены объектов страниц, сгенерированные исходным генератором, фактически были чёрным ящиком для потребителя, если не смотреть вручную в результаты сборки; это замедляло итеративное изучение поддерживаемого контракта объектов страниц.
- Предложения:
  - Заменить заглушку headless-хуков на компилируемый вариант по умолчанию для шаблонов Avalonia или хотя бы включить в комментарий каркаса точные пространства имён и типы.
  - Добавить рядом с разделом быстрого старта для headless короткий раздел по устранению проблем:
    - "Если вы видите `Headless session is not initialized`, свяжите `HeadlessUnitTestSession.StartNew(...)` с `HeadlessRuntime.SetSession(...)`."
  - Явно документировать, что потребителям следует задавать `AutomationProperties.Name` для интерактивных элементов, если они собираются использовать хелперы `WaitUntilName*`, а не предполагать, что `Content`/`Text` будут стабильно проецироваться сами.
  - Добавить рекомендации о том, какие проверки устойчивы в Avalonia headless:
    - предпочитать `AutomationId`, `IsEnabled`, `IsChecked` и прямые свойства контролов;
    - считать динамические доступные имена необязательными, если они не заданы явно.
  - Описать в документации для потребителей различия командной строки TUnit/MTP:
    - `dotnet test --project ...`
    - фильтрация и отладка через специфичные для раннера опции, а не через классический `--filter`.
  - Рассмотреть возможность упростить просмотр сгенерированных членов page object'ов или выводить сгенерированный `.g.cs` в предсказуемое и документированное место.

## Запись 9. Десктопный смоук-рантайм на FlaUI
- Шаг:
  - Повторно использован тот же смоук-путь из `Authoring` против десктопного рантайма.

    - `dotnet test --project tests\\SkillChat.UiTests.FlaUI\\SkillChat.UiTests.FlaUI.csproj -c Debug`

  - Рассмотреть документирование краткого порядка проверки по "зелёному пути" для потребителей:

    - затем десктопный рантайм;

## Запись 11. Signed-in smoke bootstrap без живого сервера

  - `TestHost` начал записывать `automation-state.json` и передавать в клиент сценарий запуска через переменные среды.
  - В клиенте добавлены deterministic fake services для backend, file picker, clipboard и notify-окна.
  - Проверены сборка `SkillChat.Client`, `SkillChat.AppAutomation.TestHost`, `Headless`, `FlaUI` и прогон существующего anonymous smoke в обоих рантаймах.

  - После введения launch-сценария существующий anonymous smoke продолжил проходить и в `Headless`, и в `FlaUI`.
- Шероховатости:
  - Для rich signed-in smoke у consumer нет готовой first-class модели "launch scenario with seeded app state" в самом `AppAutomation`; это пришлось проектировать вручную через env vars + JSON-файл состояния.
  - Для `FlaUI` и `Headless` один и тот же сценарий приходится передавать разными техническими путями:
    - в desktop runtime через `EnvironmentVariables` в launch options;
    - в headless runtime через явную инициализацию текущего процесса перед созданием окна.
  - Из-за этого потребителю приходится самостоятельно продумывать протокол между `TestHost` и AUT вместо использования готовой typed abstraction уровня framework.
  - Даже точечные сборки desktop-проектов на Avalonia здесь заметно медленные, поэтому итерация по bootstrap и `TestHost` ощущается тяжелее, чем по чисто authoring-слою.
- Предложения:
  - Добавить в `AppAutomation.TestHost.Avalonia` first-class поддержку launch profiles или typed scenario payload, который одинаково доступен и в headless, и в desktop runtime.
  - Дать consumer-friendly API для передачи deterministic app state без самодельного env-var протокола и ручной сериализации JSON.
  - Явно показать в документации паттерн "serverless signed-in shell smoke" для desktop-клиентов, а не только старт с пустого состояния.
  - Добавить рекомендации по организации seeded workspace для desktop UI automation, включая временные файлы, attachment fixtures и runtime-specific initialization order.

## Запись 12. Расширение selector contract для авторизованной оболочки
- Шаг:
  - Добавлены `AutomationId` для shell, профиля, настроек, composer-а сообщений, selection-toolbar, confirmation-dialog и attachment overlay.
  - В `MainWindowPage` расширен page object для signed-in областей и добавлено ручное разрешение динамических селекторов сообщений по `AutomationId`.
  - Проверены сборки `SkillChat.UiTests.Authoring` и `SkillChat.Client`.
- Что сработало:
  - `AppAutomation` хорошо ложится на сценарий, где consumer заранее проектирует стабильный контракт селекторов, а потом использует один page object для обоих рантаймов.
  - Прямые `AutomationId` для корневых панелей и action-button'ов дают предсказуемую основу для следующих smoke-сценариев.
  - Ручной `IUiControlResolver` позволяет дотянуться до динамических элементов коллекции, не разрывая общую authoring-модель.
- Шероховатости:
  - Для коллекционных элементов вроде сообщений у consumer нет first-class паттерна уровня page object, аналогичного `[UiControl]`, но с параметром; динамические селекторы пришлось поднимать вручную через `UiControlDefinition` и `IUiControlResolver`.
  - В реальном desktop UI часто есть парные элементы "active/inactive" с разной видимостью, и framework никак не подсказывает стратегию именования для таких случаев; это приходится придумывать вручную, чтобы селекторы не конфликтовали.
  - Для consumer-а неочевидно, какие типы `UiControlType` лучше использовать для нестандартных Avalonia-контролов вроде `ToggleSwitch`, `ItemsControl`, `ScrollViewer` и composite overlays; эту матрицу пришлось выводить эмпирически.
- Предложения:
  - Добавить в документацию раздел про динамические селекторы: как оформлять элементы коллекций, как именовать их и как лучше сочетать `AutomationId` с ручным `UiControlDefinition`.
  - Показать в шаблонах пример page object с параметризованным resolver helper для повторяющихся сущностей вроде `MessageItem_{id}` или `Row_{key}`.
  - Опубликовать компактную таблицу соответствия `Avalonia control -> рекомендуемый UiControlType` для нестандартных и составных контролов.
  - Добавить рекомендации по контракту селекторов для парных visible/invisible кнопок и panel-host/root элементов, чтобы потребители меньше гадали над схемой именования.
