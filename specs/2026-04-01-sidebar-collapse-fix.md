## 1. Overview / Цель
Исправить левую панель в Avalonia-клиенте так, чтобы верхняя кнопка реально сворачивала её до компактной ширины и освобождала дополнительное место для области сообщений.

## 2. Текущее состояние (AS-IS)
- Shell главного окна описан в `SkillChat.Client/Views/MainWindow.xaml`.
- Ширина первой колонки задаётся через `ColumnDefinition Width="{Binding SidebarWidth}"`.
- Переключение состояния идёт через `MainWindowViewModel.ToggleSidebar()` и свойства `IsSidebarExpanded` / `SidebarWidth` в `SkillChat.Client.ViewModel/MainWindowViewModel.cs`.
- По наблюдаемому поведению колонка остаётся шириной `*`, поэтому левая часть окна фактически занимает около половины ширины даже после нажатия кнопки.
- Существующие smoke-тесты проверяют только кликабельность кнопки и возврат навигации, но не валидируют фактическое изменение layout.

## 3. Проблема
Binding ширины на `ColumnDefinition` в Avalonia shell не обеспечивает фактическое применение `SidebarWidth`, поэтому команда сворачивания меняет состояние `ViewModel`, но не меняет реальный layout окна.

## 4. Цели дизайна
- Разделение ответственности: логика состояния остаётся во `ViewModel`, применение ширины переносится на визуальный элемент, который корректно участвует в binding.
- Повторное использование: не вводить отдельные обходные механизмы или дублирующее состояние.
- Тестируемость: добавить headless regression-тест, который проверяет изменение ширины сайдбара.
- Консистентность: сохранить текущий UX, automation id и поведение навигации.
- Обратная совместимость: существующие команды и свойства `MainWindowViewModel` остаются без изменения контракта.

## 5. Non-Goals (чего НЕ делаем)
- Не перерабатываем всю структуру главного окна.
- Не меняем визуальный дизайн кнопок и содержимого сайдбара.
- Не добавляем новые пользовательские настройки ширины панели.
- Не трогаем профиль/настройки/чат кроме их зависимости от освобождаемого пространства.

## 6. Предлагаемое решение (TO-BE)
### 6.1 Распределение ответственности
- `SkillChat.Client/Views/MainWindow.xaml` -> сделать первую колонку `Auto`, а ширину привязать к визуальному контейнеру сайдбара, который получает `DataContext`.
- `SkillChat.Client.ViewModel/MainWindowViewModel.cs` -> сохранить существующую модель состояния (`IsSidebarExpanded`, `SidebarWidth`, `ToggleSidebar()`), при необходимости без изменения API.
- `tests/SkillChat.UiTests.Headless/*` -> добавить regression-тест, который проверяет фактическое уменьшение ширины панели после нажатия `SidebarToggleButton`.

### 6.2 Детальный дизайн
- Заменить binding ширины на `ColumnDefinition` на layout, где колонка определяется как `Auto`, а `Border`/контейнер сайдбара имеет `Width="{Binding SidebarWidth}"`.
- Сохранить все текущие `AutomationId`, чтобы не ломать существующие UI smoke-тесты и внешние селекторы.
- Добавить headless UI test с проверкой:
  - исходная ширина сайдбара больше компактной;
  - после одного клика ширина уменьшается до компактного состояния;
  - ширина области сообщений после этого увеличивается.
- Если для headless проверки удобнее читать `Bounds.Width` реальных Avalonia-контролов, тест может работать напрямую с `MainWindow` и named controls/layout tree, без смены продуктового поведения.

## 7. Бизнес-правила / Алгоритмы (если есть)
- Expanded sidebar: ширина соответствует `SidebarWidth` для `IsSidebarExpanded = true`.
- Collapsed sidebar: ширина соответствует `SidebarWidth` для `IsSidebarExpanded = false`.
- Один клик по `SidebarToggleButton` обязан менять фактическую ширину сайдбара и перераспределять доступную ширину окна в пользу `ChatAreaRoot`.

## 8. Точки интеграции и триггеры
- Триггер: `SidebarToggleButton` -> `ToggleSidebarCommand`.
- Пересчёт layout должен происходить через обычный Avalonia binding на визуальном контейнере.
- Regression-проверка запускается в `tests/SkillChat.UiTests.Headless`.

## 9. Изменения модели данных / состояния
- Новых persisted полей нет.
- Состояние `IsSidebarExpanded` и вычисляемое `SidebarWidth` сохраняются как есть.
- Изменение касается только способа применения ширины в XAML и тестов.

## 10. Миграция / Rollout / Rollback
- Миграция данных не требуется.
- Rollout: обычный merge bugfix.
- Rollback: вернуть XAML layout и тесты к предыдущему состоянию, если выяснится несовместимость с текущим shell.

## 11. Тестирование и критерии приёмки
- Acceptance Criteria:
  - при старте signed-in shell левая панель отображается в expanded-состоянии;
  - после нажатия `SidebarToggleButton` фактическая ширина панели уменьшается до compact-состояния;
  - область сообщений после этого становится шире, чем до клика;
  - повторное раскрытие панели продолжает работать как раньше;
  - существующие automation id не меняются.
- Какие тесты добавить/изменить:
  - новый headless UI regression-тест на измерение ширины сайдбара и области сообщений;
  - при необходимости сохранить существующий smoke-тест без логических изменений.
- Команды для проверки:
  - `dotnet build`
  - `dotnet test`
  - при ускоренной итерации: `dotnet test --project tests/SkillChat.UiTests.Headless/SkillChat.UiTests.Headless.csproj`

## 12. Риски и edge cases
- Headless layout может требовать явного ожидания завершения layout pass после клика.
- Изменение на `Auto` для первой колонки не должно сломать минимальную ширину окна и выравнивание заголовка.
- Если существующие smoke-тесты завязаны на скрытие/появление навигационных элементов, нужно убедиться, что сохранён старый UX после переноса binding.

## 13. План выполнения
1. Добавить regression-тест, который воспроизводит баг через реальную ширину сайдбара в headless runtime.
2. Изменить layout `MainWindow.xaml`, чтобы binding ширины применялся к визуальному контейнеру, а не к `ColumnDefinition`.
3. Прогнать targeted UI test, затем полный `dotnet build` и `dotnet test`.
4. Сделать post-EXEC review и при необходимости поправить обнаруженные регрессии.

## 18. Альтернативы и компромиссы
- Вариант: обновлять ширину колонки из code-behind через обработчик `PropertyChanged`.
- Плюсы:
  - можно оставить текущую разметку почти без изменений.
- Минусы:
  - лишняя imperative-логика в окне;
  - больше риск расхождения между `ViewModel` и XAML.
- Почему выбранное решение лучше в контексте этой задачи:
  - перенос binding на обычный визуальный элемент проще, прозрачнее и лучше соответствует MVVM/Avalonia binding model.
